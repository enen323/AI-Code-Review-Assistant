# AI Code Review Assistant — 工程目录架构规范

> 基于 **Spring Boot 3.4 + Spring AI + MyBatis-Plus + Maven** 技术栈的目录架构决策记录。
> 维护者：项目团队 · 更新日期：2026-06-13

## 1. 框架推荐目录（来自 Maven / Spring Boot 约定）

| 目录 | 来源 | 用途 |
|------|------|------|
| `src/main/java/` | Maven Standard Directory Layout | 生产 Java 源码 |
| `src/main/resources/` | Maven Standard Directory Layout | 配置文件、静态资源、模板、数据库迁移脚本 |
| `src/test/java/` | Maven Standard Directory Layout | 单元测试 / 集成测试源码 |
| `src/test/resources/` | Maven Standard Directory Layout | 测试专用配置文件 |
| `src/main/resources/application.yml` | Spring Boot 约定 | 主应用配置（默认 profile） |
| `src/main/resources/application-{profile}.yml` | Spring Boot 约定 | 按 profile 拆分的环境配置（local / ci） |
| `src/main/resources/db/migration/` | Flyway 约定 | 数据库版本迁移脚本（`V{序号}__{描述}.sql`） |
| `target/` | Maven 构建产物 | 编译输出、打包结果，不纳入版本控制 |

> **说明：** Maven 的 `src/main/java` 下默认包路径（`com.ai.code.review`）由项目 `groupId` + `artifactId` 推导得出，Spring Boot 的主类 `AiCodeReviewAssistantApplication` 放置于此包的根位置，以保证 `@ComponentScan` / `@EntityScan` 的自动扫描范围覆盖所有子包。

## 3. 为什么这样自定义 —— 设计原则

### 3.1 按「职责维度」纵向切分，不按「技术层」横向切分

传统分层架构（controller / service / repository / entity）在此项目中被 **有意识地放弃**，转而采用 **按业务能力垂直划分** 的包结构。理由：

| 传统分层的问题 | 本项目的方案 |
|---|---|
| 一个功能横跨 4+ 包，内聚性差 | 一个 Agent 的所有代码在 `agent/xxx/` 一个包内 |
| 新增功能要在每层加类，心智负担高 | 新增维度只需加 `agent/xxx/` 一个包 |
| 不利于微服务化拆分 | 每个 Agent 包天然是 Future Domain 边界 |

### 3.2 多 Agent 架构决定目录拓扑

项目核心概念是 **多智能体审评系统**：每个审评维度（架构 / 风格 / 逻辑 / 安全）是一个独立 Agent。目录结构直接反映这一架构：

- `agent/` — 审评智能体，多态实现 `ReviewAgent` 接口
- `orchestration/` — 编排多个 Agent 的执行顺序与并发策略
- `aggregation/` — 汇总各 Agent 结果，去重、排序、评级
- `context/` — 输入（Git Diff 解析、PR 上下文获取）
- `trigger/` — 入口（Webhook + 内部事件）
- `report/` — 输出（生成报告 + 发布到 PR）
- `tool/` — Agent 调用的第三方静态分析工具封装

数据流向：`trigger → context → orchestration → agent* → aggregation → report`

### 3.3 模型统一放置

虽然每个 Agent 理论上可以有自己的 request/response 模型，但当前阶段各 Agent 的输入输出结构高度相似（`CodeContext` in → `List<ReviewResult>` out），因此将领域模型集中在 `model/` 包下，避免重复定义。后续当某 Agent 的需求差异足够大时，再考虑在 `agent/xxx/` 包内增加子模型。

### 3.4 memory 独立成包

`memory/` 是系统的「长期记忆」模块，包含向量数据库操作（`MemoryService`）、反馈学习（`FeedbackCollector`、`FalsePositiveFilter`）、统计（`RuleStats`）。它不是某个 Agent 的专属，而是被所有 Agent 和编排层共同使用，因此提升到顶级包。

### 3.5 common 预留

`common/` 当前仅有 `package-info.java`，为未来抽取跨模块工具类、常量、异常定义预留空间。**不在此目录放置业务逻辑**，只放纯基础设施工具（字符串处理、日志封装、自定义异常等）。

### 3.6 config 集中化管理

Spring 配置类统一放在 `config/` 下，而非分散在各包中。便于快速定位 `@Bean`、`@ConfigurationProperties`、`@ConditionalOnProperty` 等配置逻辑。`application.yml` 中配置项按区域分组并注释，与 `config/` 中的 `@ConfigurationProperties` 类一一对应。

## 4. 偏离框架推荐写法的原因

### 4.1 放弃传统三层架构（Controller → Service → DAO）

**Spring Boot 官方推荐** 的分层模式通常是 controller / service / repository / entity 四层。本项目 **选择不遵守**。

**原因：** 多层架构适用于 CRUD 密集的业务系统。本项目的核心是「多 Agent 审评管道」，每个 Agent 是一个独立的 min-app，有自己的输入处理、LLM 调用、工具集成。若强行套三层架构，一个 Agent 的代码会散落在 controller（入口）、service（逻辑）、repository（持久化）、entity（模型）中，可维护性反降。此决策在 MVP 阶段已验证有效。

**何时回归：** 当出现纯 CRUD 模块（如后台管理、用户系统）时，应当在该模块内采用三层架构，而非在整个项目层面推行。

### 4.2 无独立 controller 层

传统 Spring Boot 项目会在 `controller/` 包下放所有 REST 端点。本项目只有一个入口端点 `WebhookController`，放在 `trigger/` 下。

**原因：** 系统不是 RESTful CRUD 应用，而是 **事件驱动的后台处理管道**。外部入口是 GitHub Webhook（单一 URL），内部靠 `ApplicationEventPublisher` / `@EventListener` 串联。将 Controller 按业务归属放在 `trigger/` 比单独一个 `controller/` 包更有意义。

### 4.3 工具层直接放在顶级包

`tool/` 既非 agent 的子包，也非 service 的子包，而是顶级包。

**原因：** `ArchUnitTool`、`CheckstyleTool`、`SpotBugsTool` 是 **被多个 Agent 共享** 的静态分析能力。放置在顶级包明确其「跨 Agent 共享」的定位，若放在某个 Agent 下会暗示所有权，引起争议。

### 4.4 部分文件在根包下

`AiCodeReviewAssistantApplication.java` 位于 `com.ai.code.review` 根包，这也是 Spring Boot 约定（`@SpringBootApplication` 自动扫描根包）。**此为合规，非偏离。**

## 5. 后续新增模块文件放置规则

### 5.1 新增审评维度（最常见场景）

**问题：** 要加一个「性能审评」Agent，文件放哪？

**规则：**

```
agent/performance/
├── package-info.java           # 必须 — 包文档
└── PerformanceAgent.java        # 必须 — 实现 ReviewAgent 接口
```

**额外步骤：**
1. `PerformanceAgent` 实现 `ReviewAgent` 接口，标注 `@Component`，`getAgentType()` 返回 `"performance"`
2. 若该 Agent 需要专属模型，在 `model/` 下扩展现有类，或在 `agent/performance/` 包内新增私有模型（见 5.4 决策
3. 在 `orchestration/` 中注册进 Agent 列表（如 `OrchestratorAgent` 中注入）
4. 测试放在 `src/test/java/.../agent/performance/` 下

### 5.2 新增静态分析工具

**规则：** 放在 `tool/` 包下，命名格式为 `{工具名}Tool.java`

```
tool/
├── SpotBugsTool.java    # 已有
├── CheckstyleTool.java  # 已有
├── ArchUnitTool.java    # 已有
└── PMDTool.java         # 新增例子
```

### 5.3 新增集成能力

**规则：** 按外部系统名称命名包，放在顶级包下

```
gitlab/                    # 新增 GitLab 集成
├── GitLabClient.java
└── GitLabWebhookPayload.java
```

**原则：** 如果是「现有系统换同类替代品」，放在同包下或用 profile 隔离；如果是「全新的外部系统」，建立新顶级包。

### 5.4 包内扩展现有类的决策树

```
需要新增的类 → 是给特定 Agent 用的，还是全局共享？
├── 全局共享（被 ≥2 个模块使用）→ 放在 model/ / tool/ / memory/ 等全局包
└── 仅被 1 个 Agent 使用
    ├── 简单的枚举 / DTO → 直接在该 Agent 包内新建
    └── 足够复杂的子模块 → 在该 Agent 包下建子包
```

### 5.5 测试目录必须镜像

任何 `src/main/java` 下的新包/文件，必须在 `src/test/java` 下有对应测试。测试类命名为 `{被测类名}Test`，放在镜像路径下。

### 5.6 数据库迁移

新增表或修改表结构：
```
src/main/resources/db/migration/V{版本号}__{描述}.sql
```
遵循 Flyway 命名规范：`V` 大写 + 数字版本 + 双下划线 + 英文描述。

## 6. 最大化利用框架

### 6.1 Spring Boot

| 框架能力 | 本项目的使用 | 改进空间 |
|----------|------------|----------|
| `@SpringBootApplication` | 正确使用在主类 | OK |
| `@ComponentScan` 自动扫描 | 依赖默认扫描 | 需注意主类在包根位置，新增顶级包不要超出 `com.ai.code.review` |
| `@ConfigurationProperties` | `WebhookConfig` 使用了 | `AIConfig` 可改为 `@ConfigurationProperties` 绑定而非 `@Value` |
| `@EventListener` / `ApplicationEventPublisher` | `ReviewTaskEvent` + `ReviewTaskEventListener` 已使用 | OK，继续用事件驱动解耦管道 |
| `@ConditionalOnProperty` | 暂未使用 | 建议：当新增工具类可动态启停时使用 |
| `application-{profile}.yml` | local / ci 两个 profile | OK |
| `spring-boot-starter-actuator` | 仅暴露 health | 可增加 readiness / liveness 探针用于 K8s |
| `spring-boot-starter-validation` | 依赖引入 | 建议对 Webhook 请求体加 `@Valid` 校验 |

### 6.2 Spring AI

| 框架能力 | 使用现状 | 改进建议 |
|----------|---------|---------|
| `ChatClient` | 各 Agent 使用中 | OK，轻量且可测试 |
| `spring-ai-pgvector-store` | 引入依赖 | 建议用于记忆/反馈的向量语义检索 |
| `spring-ai-test` | 依赖引入，未深入使用 | 建议用 `MockClient` 替代真实的 LLM 调用做测试 |
| Tool Calling | `ArchUnitTool` 通过 Spring AI Tool 注入 | `CheckstyleTool` / `SpotBugsTool` 也应改为 Spring AI Tool 方式注入而非直接进程调用 |

### 6.3 MyBatis-Plus

| 框架能力 | 使用现状 | 改进建议 |
|----------|---------|---------|
| BaseMapper | `FeedbackRecordMapper` / `RuleStatsMapper` 继承了 | OK |
| 自动分页 | 暂未使用 | 当 feedback 数据量大时启用 |
| 条件构造器 `QueryWrapper` / `LambdaQueryWrapper` | 暂未使用 | 替代手写 SQL |

### 6.4 Flyway

| 框架能力 | 使用现状 | 改进建议 |
|----------|---------|---------|
| 版本迁移 | V1/V2 已使用 | OK，约定大于配置 |

### 6.5 GitHub API (kohsuke)

| 能力 | 使用现状 |
|------|---------|
| `GHRepository` / `GHPullRequest` / `GHIssueComment` | 在 `GitHubClient` 和 `PRCommentPublisher` 中使用 |

### 6.6 延续性建议

1. **保持多 Agent 的 SPI 风格**：新增 Agent 只需实现 `ReviewAgent` + 注册为 `@Component`，编排层自动发现，无需修改 `OrchestratorAgent` 代码（用 `List<ReviewAgent>` 注入）。
2. **配置外部化**：所有环境差异项（URL、密钥、超时）已通过 `application-{profile}.yml` + `secrets` 文件 + 环境变量管理，新增集成能力时保持此模式。
3. **事件驱动管道**：审评触发 → 任务创建 → Agent 执行 → 结果聚合 → 报告发布，每一步尽量通过 `ApplicationEventPublisher` 解耦，便于异步化和可观测性接入。
4. **测试策略**：Agent 逻辑用 Spring AI `MockClient`；编排层用 `@SpringBootTest` + mock Agent；工具集成层用真实工具但 mock 文件系统。
5. **模块边界**：顶级包之间的依赖方向必须单向 —— `trigger → context → orchestration → agent / tool → aggregation → report`，禁止反向依赖。

## 附录 B：术语表

| 术语 | 含义 |
|------|------|
| Agent | 审评智能体，一个独立的审评维度，实现 `ReviewAgent` 接口 |
